Cumulative Table Design

A pattern for keeping rolling user history cheaply: scan only today’s raw events, combine them with yesterday’s cumulative snapshot, and write today’s new snapshot.

1 dayraw scan per run
30days stored in arrays
fast7d/30d metrics

Business Scenario

A consumer app wants daily, weekly, and monthly active users plus 7-day and 30-day action counts. Raw events are huge, so scanning the last 30 days every day gets expensive.

Raw Dataevents: one row per like, comment, share, view, play, click, etc.
Daily Summaryactive_users_daily: one row per user per day.
Memory Tableactive_users_cumulated: arrays holding recent user history.

End Goal

Give analysts a table where DAU, WAU, MAU, and rolling 7/30-day actions are already easy to compute without repeatedly scanning raw event history.

End UsersAnalytics, growth, product, lifecycle marketing, ML features.
TimingUsually updates once per daily partition after today’s events are complete.
OutputOne user snapshot per day with arrays and precomputed metrics.

Practical Use Cases

  • Engagement: DAU/WAU/MAU by user.
  • Rolling counts: likes, comments, plays, purchases.
  • Lifecycle: recently active, dormant, returning users.
  • ML features: compact recent behavior vectors.

How It Saves Money

  • Less scanning: read today plus yesterday, not 30 raw days.
  • Fast rollups: 7d/30d metrics are array sums.
  • Stable cost: daily work stays bounded as raw history grows.
  • Cheaper dashboards: consumers query a prepared table.

When To Use It

  • You need rolling windows for large event data.
  • Daily partition processing is acceptable.
  • Your entity grain is clear: user, account, product, device.
  • Late-arriving data is controlled or handled by reprocessing.

When Not To Use It

  • Data is small enough for simple window queries.
  • You need arbitrary windows not captured by stored arrays.
  • Late events frequently rewrite old days.
  • Arrays would become huge or hard to govern.

-- TABLE 1: source table.
-- This is the messy raw event pile.
create table events (
    user_id integer,
    event_type text,
    event_timestamp timestamp,
    event_date date
);

-- TABLE 2: daily table.
-- This stores one clean row per user per day.
create table active_users_daily (
    user_id integer,
    is_active_today integer,
    num_likes integer,
    num_comments integer,
    num_shares integer,
    snapshot_date date
);

-- TABLE 3: final cumulative output table.
-- This stores today's result plus little arrays of recent memory.
create table active_users_cumulated (
    user_id integer,
    is_daily_active integer,
    is_weekly_active integer,
    is_monthly_active integer,
    activity_array integer[],
    like_array integer[],
    snapshot_date date
);

-- Step 1: Make today's small summary.
-- Like a 10-year-old version: count what each user did today.
insert into active_users_daily (
    user_id, is_active_today, num_likes, num_comments, num_shares, snapshot_date
)
select
    user_id,
    1 as is_active_today, -- If the user has any event today, they are active.
    count(*) filter (where event_type = 'like') as num_likes,
    count(*) filter (where event_type = 'comment') as num_comments,
    count(*) filter (where event_type = 'share') as num_shares,
    date '2024-01-10' as snapshot_date
from events
where event_date = date '2024-01-10' -- Only look at today's small pile.
group by user_id;

-- Step 2: Add today's number to yesterday's memory.
-- ARRAY[x] makes a tiny one-item list.
-- old_array[1:29] keeps only the first 29 old days.
-- Together: 1 new day + 29 old days = max 30 days.
insert into active_users_cumulated (
    user_id,
    is_daily_active,
    is_weekly_active,
    is_monthly_active,
    activity_array,
    like_array,
    snapshot_date
)
with today as (
    select * from active_users_daily
    where snapshot_date = date '2024-01-10'
),
yesterday as (
    select * from active_users_cumulated
    where snapshot_date = date '2024-01-09'
),
combined as (
    select
        coalesce(t.user_id, y.user_id) as user_id,
        array[coalesce(t.is_active_today, 0)] || coalesce(y.activity_array[1:29], array[]::integer[]) as activity_array,
        array[coalesce(t.num_likes, 0)] || coalesce(y.like_array[1:29], array[]::integer[]) as like_array
    from today t
    full outer join yesterday y
        on t.user_id = y.user_id -- Match the same user across today and yesterday.
)
select
    user_id,
    activity_array[1] as is_daily_active, -- First box is today.
    case when (select sum(x) from unnest(activity_array[1:7]) as u(x)) > 0 then 1 else 0 end as is_weekly_active,
    case when (select sum(x) from unnest(activity_array) as u(x)) > 0 then 1 else 0 end as is_monthly_active,
    activity_array,
    like_array,
    date '2024-01-10' as snapshot_date
from combined;

Core idea

Today’s snapshot is built from today’s daily facts plus yesterday’s memory. The table carries history forward one partition at a time.
