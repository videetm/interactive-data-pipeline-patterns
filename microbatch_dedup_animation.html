Watch raw hourly partitions become compact hourly facts, then merge in rounds until the day has one clean deduped output.

Business Scenario

Imagine an ecommerce company tracking product events from both the website and mobile app: product views, clicks, add-to-cart events, purchases, and checkout retries. Event collectors, mobile retries, and late network delivery can send the same business event more than once.

Raw Data event_source: one row per event, partitioned by date, hour, and product_name.
Duplicate Problem The same product/event can appear twice inside one hour or again in the next hour.
Why Hourly The business wants fresh daily numbers before all 24 hours are processed at once.

End Goal

Produce one clean daily table that analytics, ML, and reporting teams can trust without scanning the full raw firehose repeatedly.

1Deduped daily product events in deduped_source.
2Lower latency because each hour can be processed as soon as it lands.
3Less compute because merge rounds read compact deduped data, not raw events.

Practical Use Cases

  • Product analytics: dedupe product views, clicks, add-to-cart, and purchase events.
  • Ad tech: dedupe impressions and clicks before attribution or billing.
  • Fraud/risk: process fresh event streams without double-counting retries.
  • IoT/logs: compact repeated device pings or telemetry events by hour.

How It Saves Money

  • Less raw scanning: later merges read compact hourly outputs instead of full raw data.
  • Parallel work: hour 1 and hour 2 can dedupe while later hours are still arriving.
  • Smaller retries: if one hour fails, rerun that small partition, not the whole day.
  • Cheaper downstream: reports and ML jobs read one clean daily table.

When To Use It

  • Your raw event volume is large enough that daily full scans are expensive.
  • Duplicates can appear inside one hour and across nearby hours.
  • The business wants fresher daily data before a giant end-of-day batch finishes.
  • You have a reliable dedupe key such as event_id, order_id, request_id, or a strong composite key.

When Not To Use It

  • Your data is small and a simple daily dedupe is cheap enough.
  • You cannot define what makes two events the same real-world action.
  • Duplicates can arrive many days late and your merge window is only hourly/same-day.
  • Your team does not need lower latency and would prefer a simpler batch job.

-- TABLE 1: source table.
-- This is the messy raw event pile, partitioned by day/hour/product.
create table event_source (
    ds date,
    hour integer,
    product_name text,
    product_id bigint,
    event_type text,
    event_timestamp_epoch bigint,
    event_location text
);

-- TABLE 2: hourly intermediate table.
-- First it stores each cleaned hour.
-- Later it also stores merge partitions like hour = 12, meaning "hours 1 through 2".
create table hourly_deduped_source (
    ds date,
    hour integer,
    product_name text,
    product_id bigint,
    event_type text,
    min_event_timestamp_epoch bigint,
    max_event_timestamp_epoch bigint,
    event_locations jsonb
);

-- TABLE 3: final daily output table.
-- This is what downstream dashboards/jobs should read for the full day.
create table deduped_source (
    ds date,
    product_name text,
    product_id bigint,
    event_type text,
    min_event_timestamp_epoch bigint,
    max_event_timestamp_epoch bigint,
    event_locations jsonb
);

-- Query 1: dedupe one hour.
-- Like a 10-year-old version:
-- "Put the same product/event rows into one bucket for this hour."
insert into hourly_deduped_source (
    ds, hour, product_name,
    product_id, event_type,
    min_event_timestamp_epoch, max_event_timestamp_epoch,
    event_locations
)
select
    ds,
    hour,
    product_name,
    product_id,
    event_type,
    min(event_timestamp_epoch) as min_event_timestamp_epoch, -- earliest time we saw it
    max(event_timestamp_epoch) as max_event_timestamp_epoch, -- latest time we saw it
    jsonb_object_agg(event_location, event_timestamp_epoch) as event_locations -- keep location clues
from event_source
where ds = date '2024-01-10'
  and hour = 1
  and product_name = 'website'
group by ds, hour, product_name, product_id, event_type;

-- Query 2: merge hour 1 and hour 2.
-- FULL OUTER JOIN means:
-- keep rows only in hour 1, only in hour 2, and rows in both.
with earlier as (
    select * from hourly_deduped_source
    where ds = date '2024-01-10' and hour = 1 and product_name = 'website'
),
later as (
    select * from hourly_deduped_source
    where ds = date '2024-01-10' and hour = 2 and product_name = 'website'
)
insert into hourly_deduped_source (
    ds, hour, product_name,
    product_id, event_type,
    min_event_timestamp_epoch, max_event_timestamp_epoch,
    event_locations
)
select
    date '2024-01-10' as ds,
    12 as hour, -- pretend this partition means "merged hours 1 through 2"
    'website' as product_name,
    coalesce(e.product_id, l.product_id) as product_id,
    coalesce(e.event_type, l.event_type) as event_type,
    least(coalesce(e.min_event_timestamp_epoch, l.min_event_timestamp_epoch),
          coalesce(l.min_event_timestamp_epoch, e.min_event_timestamp_epoch)) as min_event_timestamp_epoch,
    greatest(coalesce(e.max_event_timestamp_epoch, l.max_event_timestamp_epoch),
             coalesce(l.max_event_timestamp_epoch, e.max_event_timestamp_epoch)) as max_event_timestamp_epoch,
    coalesce(e.event_locations, '{}'::jsonb) || coalesce(l.event_locations, '{}'::jsonb) as event_locations
from earlier e
full outer join later l
    on e.product_id = l.product_id
   and e.event_type = l.event_type;
-- Final step after all 24 hours have been merged.
-- Copy the final merge partition into the final daily table.
-- Like a 10-year-old version:
-- "Take the cleaned whole-day box and put it on the shelf for everyone to use."
insert into deduped_source (
    ds,
    product_name,
    product_id,
    event_type,
    min_event_timestamp_epoch,
    max_event_timestamp_epoch,
    event_locations
)
select
    ds,
    product_name,
    product_id,
    event_type,
    min_event_timestamp_epoch,
    max_event_timestamp_epoch,
    event_locations
from hourly_deduped_source
where ds = date '2024-01-10'
  and hour = 124 -- pretend this means "merged hours 1 through 24"
  and product_name = 'website';

Tables In This Pattern

event_source is the raw hourly input. hourly_deduped_source is the reusable intermediate table. deduped_source is the final daily output.

Why not only dedupe each hour?

The same product/event key can appear in hour 1 and hour 2. The merge rounds dedupe across hour boundaries without waiting to scan the whole day at once.

deduped(1-24) = merge(merge(1-16), merge(17-24))
