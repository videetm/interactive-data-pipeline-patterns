Little Book of Pipelines

A pattern for pipelines with too many upstream sources: group sources, normalize them to one shared schema, and keep a small “book” of metadata that makes jobs, data quality, and backfills easier to reason about.

10+upstream sources
1shared schema
Nsimple group jobs

Business Scenario

A company has many source systems feeding one analytics domain: billing tables, CRM tables, app events, support tickets, ad platform exports, and product catalogs. Everyone wants one usable output, but each source has different columns, quality rules, and backfill behavior.

Raw InputsMany unrelated upstream tables with different schemas.
ProblemBackfills hurt because one giant pipeline owns too much logic.
PatternGroup sources, convert to an intermediate schema, then union/write.

End Goal

Create a cleaner output table and a “little book” catalog that tells the team what each group/item is, which job owns it, and what data quality rules apply.

Output DataIntermediateDataset: identifier, typeName, properties, groupName.
Output MetadataItemGroupEnum: groups, items, thresholds, constants.
UsersData engineering, analytics, DQ dashboards, and backfill operators.

Practical Use Cases

  • Many sources: 10+ feeds producing one domain table.
  • Entity catalogs: customers, items, vendors, campaigns, products.
  • DQ dashboards: metadata-driven checks per group/item.
  • Backfills: rerun one group instead of a giant job.

How It Saves Money

  • Smaller reruns: failed group jobs can be replayed independently.
  • Less engineer time: source ownership is visible in code.
  • Reusable schema: downstream consumers stop handling every source shape.
  • Faster debugging: bad item maps to one group/job.

When To Use It

  • Many sources feed a common analytical concept.
  • You need clearer ownership boundaries.
  • Sources have different DQ thresholds or constants.
  • Backfills are frequent and painful.

When Not To Use It

  • You only have one or two sources.
  • The sources do not share a meaningful output schema.
  • The metadata catalog would become fake ceremony.
  • You need near-real-time streaming semantics.

-- TABLE 1: first source table.
-- This is CRM-shaped data.
create table crm_customers (
    ds date,
    customer_id text,
    plan text,
    region text
);

-- TABLE 2: second source table.
-- This is billing-shaped data.
create table billing_accounts (
    ds date,
    account_id text,
    mrr numeric,
    currency text
);

-- TABLE 3: little book metadata table.
-- This is the catalog that says what each item means and which group owns it.
create table item_group_book (
    group_name text,
    type_name text,
    dq_rule text
);

-- TABLE 4: final shared output table.
-- Every source gets squeezed into this same shape.
create table intermediate_dataset (
    identifier text,
    type_name text,
    properties jsonb,
    group_name text,
    ds date
);

-- The target table has one common shape:
-- identifier = the thing's ID
-- type_name = what kind of thing it is
-- properties = extra details packed into JSON
-- group_name = the owner bucket

insert into intermediate_dataset (
    identifier,
    type_name,
    properties,
    group_name,
    ds
)

-- First source: CRM customers.
-- Like a 10-year-old version:
-- "Take the CRM toy, put it into the common box shape."
select
    customer_id as identifier,
    'crm_customer' as type_name,
    jsonb_build_object(
        'plan', plan,
        'region', region
    ) as properties,
    'customer_group' as group_name,
    date '2024-01-10' as ds
from crm_customers
where ds = date '2024-01-10'

union all

-- Second source: billing accounts.
-- Same common box shape, even though the source columns are different.
select
    account_id as identifier,
    'billing_account' as type_name,
    jsonb_build_object(
        'mrr', mrr,
        'currency', currency
    ) as properties,
    'revenue_group' as group_name,
    date '2024-01-10' as ds
from billing_accounts
where ds = date '2024-01-10';
